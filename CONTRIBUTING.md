# Правила участия в проекте "Project Chronicle"

Мы рады вашему вкладу в развитие проекта! Чтобы поддерживать высокое качество кодовой базы и понятную историю изменений, мы придерживаемся строгих правил оформления коммитов.

## Conventional Commits 1.0.0

Все коммиты в этом проекте **должны** соответствовать спецификации [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/). Это соглашение о создании явных сообщений к коммитам. Оно позволяет легко автоматизировать процессы (например, генерацию списка изменений) и упрощает навигацию по истории проекта для других разработчиков.

### Структура сообщения коммита

Каждое сообщение коммита состоит из **заголовка**, необязательного **тела** и необязательного **подвала**.

---

### 1. Заголовок (Header)

Заголовок — это единственная обязательная часть сообщения. Он имеет формат: `<type>[optional scope]: <description>`

#### **Тип (`<type>`)**

Тип описывает категорию изменений, которые вы внесли. Мы используем следующие типы:

- **feat:** Новая функциональность (feature).
    - *Пример: `feat: Добавлена возможность поиска записей по дате`*

- **fix:** Исправление бага (bug fix).
    - *Пример: `fix: Исправлена ошибка, из-за которой не отображалась первая запись`*

- **refactor:** Изменение кода, которое не исправляет баг и не добавляет новую функциональность. Например, улучшение структуры или читаемости кода.
    - *Пример: `refactor: Упрощена логика чтения файла в методе listEntries`*

- **docs:** Изменения, касающиеся только документации (`README.md`, комментарии в коде и т.д.).
    - *Пример: `docs: Обновлен README с инструкциями по сборке`*

- **test:** Добавление или исправление тестов.
    - *Пример: `test: Добавлен новый тест для проверки поиска по дате`*

- **build:** Изменения, которые влияют на систему сборки или внешние зависимости (`build.gradle`, `pom.xml`, `package.json`).
    - *Пример: `build: Обновлена версия JUnit до 5.9.2`*

- **ci:** Изменения в файлах и скриптах для непрерывной интеграции (CI) (`.github/workflows/ci.yml`).
    - *Пример: `ci: Ускорен шаг сборки в GitHub Actions`*

- **style:** Изменения, не влияющие на смысл кода (пробелы, форматирование, точки с запятой и т.д.).
    - *Пример: `style: Исправлено форматирование в классе Journal`*

- **revert:** Если коммит отменяет предыдущий коммит. В теле коммита **должна** быть информация об отменяемом коммите.
    - *Пример заголовка: `revert: feat: Добавлены временные метки к записям`*
    - *Пример тела: `This reverts commit 8675309...`*

#### **Область (`[optional scope]`)**

Необязательное поле, которое указывает на часть кодовой базы, затронутую изменениями. Заключается в круглые скобки.

- *Пример: `feat(search): Реализован полнотекстовый поиск`*
- *Пример: `fix(api): Исправлена обработка некорректных ответов`*

#### **Описание (`<description>`)**

Краткое описание изменений в настоящем времени и в повелительном наклонении.

- **Правильно:** `feat: Добавить поддержку формата JSON`
- **Неправильно:** `Добавил поддержку JSON` или `Поддержка JSON была добавлена`

---

### 2. Тело (Body)

Необязательная часть, которая используется, чтобы предоставить дополнительный контекст. Отделяется от заголовка пустой строкой.

- Описывает **что** было изменено и **почему**, а не **как**.
- *Пример:* `fix: Исправлена ошибка аутентификации
  Предыдущая реализация неверно обрабатывала истекшие токены.
  Теперь при получении статуса 401 система корректно
  перенаправляет пользователя на страницу входа.`

---

### 3. Подвал (Footer)

Необязательная часть, которая используется для указания **критических изменений** (breaking changes) или для ссылки на задачи в трекерах (JIRA, GitHub Issues).

#### **Критические изменения (BREAKING CHANGE)**

Если ваши изменения нарушают обратную совместимость, это **должно** быть указано в подвале, начиная с `BREAKING CHANGE:`.

- *Пример:* `feat: Изменен формат API для записей
BREAKING CHANGE: Поле timestamp в ответе API теперь
возвращается в формате ISO 8601 вместо Unix timestamp.
Клиенты, использующие старый формат, должны быть обновлены.`

#### **Ссылка на задачи**

Если коммит связан с какой-либо задачей, укажите ее номер.

- *Пример: `Closes CHRON-404` или `Fixes #123`*

---

### Пример полного коммита

`feat(api): Добавлена пагинация для списка записей
Реализована пагинация на стороне сервера для эндпоинта /entries.
Это предотвращает загрузку тысяч записей одновременно и
улучшает производительность.
BREAKING CHANGE: Эндпоинт /entries теперь возвращает объект
с полями data и pagination вместо простого массива.
Closes #42`

**Соблюдение этих правил является обязательным для принятия вашего вклада в проект.**
